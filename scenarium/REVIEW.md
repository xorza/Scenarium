# Scenarium architecture and simplification review

## Executive summary

Scenarium has a strong top-level architecture. The authoring `Graph` is distinct from the compiled program, composite flattening is isolated, and the plan → resolve → execute phases have unusually clear ownership of structural, cache-aware, and mutable runtime state. The graph side tables and the cache state machine also encode important invariants deliberately rather than hiding them in convention.

The main weaknesses are now at subsystem seams:

- Execution reports remain flat-runtime data at the public callback boundary, although each ID now has a distinct execution-only type and an ordered installed program supplies its interpretation.
- The optional disk cache can turn a corrupt or concurrently removed blob into a failed node for one run, and its header probes perform synchronous filesystem I/O on the async worker path.
- A stable `FuncId` is also the entire computation-version key, so changing a function implementation can silently reuse values produced by the old implementation.
- Function, type, and context registration APIs permit invalid or misleading states that are rejected late—or, for several advisory fields, never consumed at all.
- The lambda ABI is six positional borrows plus a one-field `InvokeInput` wrapper, while `ContextManager` combines persistent resources with per-run cancellation, attribution, and logs.
- The compiled program's parallel span-indexed pools remove allocations but spread their invariants across flattening, type resolution, validation, planning, caching, and execution. That optimization should exist only if it has measured value.
- The folder structure is broadly sound, but `execution/mod.rs`, `cache/mod.rs`, and `executor/mod.rs` still combine distinct responsibilities despite already having directory-module homes.

The most valuable sequence is to fix execution identity and cache transparency first, then make registries and invocation state valid by construction. Data-layout and file-organization simplifications are safer after those contracts settle.

## Current flow

`Compiler` validates an authoring `Graph`, recursively flattens graph instances into an `ExecutionProgram`, resolves output types, and returns a self-contained `CompiledGraph`. `Worker` reduces batches of public `WorkerMessage`s into an intent, installs shared compiled state, acknowledges that state through its ordered report stream, plans roots, prepares resource stamps, resolves digests/cache liveness, executes surviving nodes, and reports flat-ID progress and statistics. `RuntimeCache` owns persistent node state, resident outputs, disk availability state, digest stamping, codec dispatch, hydration, persistence, reclamation, and RAM accounting.

## Batch 1 — Critical: make executions self-identifying and seed-safe

- [ ] **Return authoring addresses from the public reporting boundary.** The ordered `Installed(Arc<CompiledGraph>)` report fixes program correlation for Scenarium's serial worker, and `ExecutionNodeId`/`ExecutionInputPort` now prevent flat identities from being confused with authoring `NodeId`/`InputPort`. However, `RunProgress`, almost every field of `ExecutionStats`, and worker errors still expose execution identities (`src/execution/report.rs`, `src/execution/stats.rs`, `src/worker/protocol.rs`), leaving downstream consumers responsible for projection. Map runtime results into `NodeAddress` plus named input/output address types before invoking the public callback, and keep flat-only statistics private to execution.

- [ ] **Validate event seeds as strictly as exact node seeds.** Exact `ExecutionNodeId` node seeds return `Error::NodeSeedNotFound` when absent, but public `WorkerMessage::InjectEvents` accepts `EventRef`s and `collect_roots` directly indexes both the node map and event slice (`src/execution/plan/mod.rs`, `src/worker/protocol.rs`). One malformed, out-of-range, or stale injected event can panic the worker task. Validate its execution identity and event index with structured errors, or make injection crate-private if only the internal event loop should use it. Cover missing nodes, out-of-range indices, and events queued across a program replacement.

## Batch 2 — Critical: make disk caching transparent, versioned, and non-blocking

- [ ] **Treat every disk-cache read failure as a miss within the same execution request.** Resolution accepts a header-only disk hit and cuts its producer cone, but hydration can later delete the bad blob and return `false`; the consumer then receives `RunError::InputLoadFailed` and is skipped until a future run recomputes the producer (`src/execution/cache/mod.rs:460-502`, `src/execution/cache/mod.rs:504-566`, `src/execution/resolve/mod.rs:164-198`, `src/execution/executor/mod.rs:186-215`, `src/execution/mod.rs:169-174`). An optional cache should not change whether a valid graph succeeds. After the reverse sweep, pre-hydrate only reused nodes on the actual external/running frontier; invalidate failures and repeat resolution so their upstream cones become live. Retain a bounded retry for a file disappearing between preflight and consumption. Delete `InputLoadFailed` once cache corruption, truncation, deletion, codec rejection, and concurrent replacement all produce the same result as a cold run in that request.

- [ ] **Move all cache presence and coverage probes off the async worker thread.** `BlobTarget::coverage` opens and reads a file synchronously, including metadata, and is called from synchronous reuse resolution, reclamation, and the async store path (`src/execution/disk_store/mod.rs:57-63`, `src/execution/disk_store/mod.rs:146-151`, `src/execution/disk_store/mod.rs:175-198`, `src/execution/cache/mod.rs:477-498`, `src/execution/cache/mod.rs:613-630`). This contradicts the otherwise careful use of `spawn_blocking` for body reads, writes, and resource stamping, and scales to one blocking syscall sequence per disk-backed node. Build a per-run `DiskAvailability` snapshot in one blocking task before cache-aware resolution, update it after writes/deletes, and make resolver/reclamation consult only memory. Validate responsiveness with deliberately slow filesystem operations and assert that no `std::fs` probe remains reachable from the async worker outside a blocking task.

- [ ] **Add an explicit function implementation revision to the content key.** `Func` carries stable identity and behavior but no implementation/cache revision, while `node_digest` hashes only the global domain, `FuncId`, output signature, and inputs; the module documentation explicitly acknowledges that changed logic under the same ID can serve an old value (`src/node/definition.rs:185-212`, `src/execution/digest/mod.rs:15-21`, `src/execution/digest/mod.rs:236-290`). Stable IDs are persisted graph identity and should not be repurposed as release versions. Add a required or consciously defaulted `cache_revision` to executable function registration and fold it beside `FuncId`; make replacement of a registered function require an explicit revision decision. Validate equal implementation/revision hits, revision-only misses, downstream re-keying, and RAM plus disk behavior across a library rebuild.

- [ ] **Version custom codecs in the outer blob key instead of discovering incompatibility during lazy decode.** `TypeEntry` records only an optional codec, `outputs_decodable` checks only codec presence, and the digest documentation requires a global cache-domain bump for a breaking codec change (`src/library.rs:44-55`, `src/execution/disk_store/mod.rs:103-112`, `src/execution/digest/mod.rs:35-36`). A codec such as Lens's image codec embeds its own version, but rejection then follows the one-run failure path above (`../lens/src/image/codec/mod.rs:18-25`, `../lens/src/image/codec/mod.rs:82-91`). Give each codec registration a format revision and add a codec fingerprint to `BlobTarget`/the outer header. Keep it separate from the computation digest so a serialization change invalidates disk bytes without invalidating valid RAM values or unrelated downstream computation caches. Validate old/new codec revisions, static-only outputs, mixed custom outputs, and malformed bodies.

## Batch 3 — High: make registries hold valid states only

- [ ] **Require executable lambdas before a function can enter `Library`.** `Func` derives `Default`, `FuncLambda` has a public `None` state, `Func::validate` does not reject it, and the executor handles the resulting host configuration bug as a per-node runtime error (`src/node/definition.rs:185-223`, `src/node/definition.rs:312-349`, `src/node/lambda.rs:85-128`, `src/execution/executor/mod.rs:334-347`). Make the registered `Func` valid by construction—prefer `Func::new(id, name, lambda)` or a builder that can only produce a `Func` after receiving a lambda—then turn `FuncLambda` into an `Arc` newtype and delete `RunError::MissingLambda` plus its executor branch. Apply the same invariant to `EventLambda`/`FuncEvent`, so an advertised event always has an implementation. Validate registration-time rejection or compile-time impossibility of missing implementations and preserve panic isolation only for lambdas that actually panic while running.

- [ ] **Make registry collision behavior explicit and consistent.** Function insertion and shared-graph insertion silently replace an existing identity, type registration panics on a duplicate, and `Library::merge` inherits all three policies without naming which registries are overlays (`src/library.rs:133-175`, `src/library.rs:214-225`). Silent function replacement is especially dangerous once `FuncId` participates in persistent cache identity. Make ordinary registration reject duplicates for every registry, provide an explicitly named replacement operation only where live editing needs it, and make `merge` either reject all collisions or accept a named conflict policy. Validate duplicate IDs across composed built-in libraries, intentional graph replacement, and function replacement with the revision rule from Batch 2.

- [ ] **Replace `TypeDecl` plus optional runtime attachments with one sum type that cannot represent enum codecs or stampers.** `TypeEntry` exposes `decl`, `codec`, and `stamper` publicly, then uses a fallible `validate` solely to reject combinations its structure permits (`src/library.rs:13-64`, `src/library.rs:67-112`, `src/library.rs:168-180`). Define private-field variants such as `TypeEntry::Custom(CustomTypeEntry)` and `TypeEntry::Enum(EnumTypeEntry)`; put codec, stamper, and codec revision only on the custom variant, and expose focused queries from a private type registry. This deletes the parallel `TypeDecl` projection and the logic-error `anyhow::Result` path. Validate construction, enum lookup, custom stamping/codec lookup, duplicate registration, and library merge.

- [ ] **Give disk caching a narrow codec registry instead of the entire graph/function library, and make disabled storage an outer option.** `DiskStore` holds `Arc<Library>` solely to find custom codecs, while its `disk_root: Option<PathBuf>` creates a disabled `DiskStore` state that every target lookup rechecks (`src/execution/disk_store/mod.rs:29-40`, `src/execution/disk_store/mod.rs:73-112`, `src/execution/codec.rs:80-125`). Extract an immutable `CodecRegistry`/type-runtime view from `Library`, construct an enabled `DiskStore` with a required root, and let `RuntimeCache` own `Option<DiskStore>`. Graph or function-library edits would then stop forcing disk-store replacement when codec registrations are unchanged, and storage code could no longer reach unrelated graphs, funcs, or editor metadata. Validate enabling/disabling, root swaps, type-registry swaps, and persistence across graph-only library changes.

## Batch 4 — High: simplify the invocation and context APIs

- [ ] **Replace the six-argument lambda ABI with one named invocation object and delete the one-field `InvokeInput`.** `AsyncLambdaFn` takes six positional mutable/shared borrows, `InvokeInput` only wraps `DynamicValue`, the macro duplicates four arity-heavy patterns, and the executor allocates/refills wrappers before unpacking the same six values at the call site (`src/node/lambda.rs:42-83`, `src/node/macros.rs:1-25`, `src/execution/executor/mod.rs:119-128`, `src/execution/executor/mod.rs:186-215`, `src/execution/executor/mod.rs:430-448`). Introduce `Invocation<'a>` with public named fields and use `&mut [DynamicValue]` directly for inputs; make a lambda accept only that value. The async helper macro can then collapse to at most the move/non-move capture cases, and ordinary callers can destructure only what they use. Validate moving the last custom input without cloning, output demand, node/event state, context access, cancellation, and every downstream function library.

- [ ] **Split persistent typed resources from per-run attribution, logs, and cancellation.** `ContextManager` owns a cross-run type-erased resource store and also mutable `current_node`, `logs`, and `cancel`, which the executor resets and rewrites around every invocation; `CustomValueCodec::encode` consequently receives the full run-management object merely to access a device context (`src/runtime/context.rs:23-37`, `src/runtime/context.rs:39-79`, `src/execution/executor/mod.rs:286-291`, `src/execution/executor/mod.rs:417-420`, `src/execution/executor/mod.rs:537-540`, `src/execution/codec.rs:31-39`). Keep a persistent `ContextStore` in `Executor`, put node ID/cancel/log sink in the per-call `Invocation`, and pass only `&mut ContextStore` plus any explicit cancellation need to codecs. This gives each lifecycle one owner and stops disk serialization from depending on node logging state. Validate context reuse across runs, per-run log isolation, cancellation visibility, and codec access to the same persistent resource.

- [ ] **Make context lookup statically typed and delete unused advisory context fields.** `ContextType` is untyped, so `ContextManager::get<T>(&ContextType)` can request the wrong `T` and only detects it by panicking after a downcast; its public `description` is never read, and `Func::required_contexts` plus the `.context(...)` builder have production writers but no production reader (`src/runtime/context.rs:16-21`, `src/runtime/context.rs:105-137`, `src/node/definition.rs:206-212`, `src/node/definition.rs:302-305`). Replace it with `ContextKey<T>` whose constructor and `ContextStore::get` preserve `T`, key the erased backing map by the private ID, and remove `description`, `required_contexts`, and the builder until a real consumer owns their semantics. Validate that a mismatched type is impossible to compile and that Lens's vision context remains lazily constructed and reused.

- [ ] **Delete the unenforced `Func::uncacheable` UI flag.** The field and builder claim a function manages its own cache, but no Scenarium execution path reads them; the only current producer is the output-less `RunSinks` special node, while Darkroom alone interprets the flag as whether to show controls (`src/node/definition.rs:192-202`, `src/node/definition.rs:247-260`, `src/elements/run_sinks.rs:25-45`, `../darkroom/src/gui/scene.rs:268-316`). Remove the field and derive UI capability from enforceable facts such as output arity, purity, and codec availability. If self-caching functions later need a core contract, add an engine-enforced cache capability enum rather than an editor-only boolean with a runtime-sounding name. Validate the cache controls for sinks, graph/boundary nodes, pure custom outputs with and without codecs, and ordinary impure functions.

## Batch 5 — Medium: simplify compiled data and align files with responsibilities

- [ ] **Replace the span-indexed parallel program pools with node-local immutable slices unless a benchmark proves the arena layout is material.** An `ExecutionNode` stores three `Span`s while inputs, events, output types, and pin flags live in separate global vectors; this requires `OutputIdx`, `OutputColumn`, a `Pools` adapter, running `u32` counters, an output-owner reconstruction pass, and cross-pool range validation (`src/execution/program.rs:20-39`, `src/execution/program.rs:68-140`, `src/execution/flatten/mod.rs:53-105`, `src/execution/flatten/mod.rs:285-329`, `src/execution/program.rs:177-225`, `src/execution/validate.rs:40-65`). Store boxed input/event/output slices on each compiled node, with one `ExecutionOutput { data_type, pinned }`, and group per-run demand/readers under that node's run state. This removes several parallel-length invariants and most index conversion code at the cost of compile-time per-node allocations. Compare compile time, repeated-run allocations, and plan/execute throughput on representative large graphs; retain the arena only if the measured result justifies its pervasive complexity.

- [ ] **Move engine orchestration, shared columns, and public errors out of `execution/mod.rs`.** The module root currently defines pool helpers and map aliases, public execution errors, run seeds, and the entire `ExecutionEngine` lifecycle in addition to declaring every submodule (`src/execution/mod.rs:20-124`, `src/execution/mod.rs:126-233`, `src/execution/mod.rs:235-371`). Create canonical `execution/error.rs`, `execution/run.rs` (seeds/request state), and `execution/engine.rs`; keep `mod.rs` to module declarations and only minimal shared vocabulary. Publish external names only through `lib.rs`, following the repository's no-internal-re-export rule. Validate unchanged public error formatting, worker behavior, and module visibility.

- [ ] **Split cache value state from cache policy, and executor frame mechanics from outcome reporting.** `cache/mod.rs` defines coverage encoding, snapshots, the value-state machine, mutable invocation slots, disk policy, digest stamping, hydration, persistence, reclamation, and RAM reports in one file (`src/execution/cache/mod.rs:31-288`, `src/execution/cache/mod.rs:296-655`). `executor/mod.rs` likewise combines input hydration/reclamation, the run loop, outcome state, and statistics projection (`src/execution/executor/mod.rs:41-240`, `src/execution/executor/mod.rs:242-543`, `src/execution/executor/mod.rs:545-640`). After the earlier contracts settle, move slot/coverage types to `cache/slot.rs`, keep tier policy in `cache/mod.rs`, move `ExecutionFrame` to `executor/frame.rs`, and move `NodeOutcome` plus stats projection to `executor/outcome.rs`. Use canonical submodule imports rather than internal re-exports. Validate cache state transitions, exact stats, and the full worker suite after each move.

## Batch 6 — Low: reduce the public and dependency surface

- [ ] **Hide the raw worker protocol behind semantic worker operations.** `Worker` publicly exposes only `send`/`send_many`, forcing callers to construct internal coordination messages such as `Update`, `Run { seeds }`, `StartEventLoop`, and `Sync { oneshot }`; nothing in the public types prevents a run command from targeting stale installed state (`src/worker/mod.rs:23-85`, `src/worker/protocol.rs:11-35`, `src/worker/batch.rs:24-94`). Make the message enum private and expose explicit install, run, store change, loop stop, synchronization, cancellation, and shutdown methods with a documented single-producer FIFO contract. Validate every batch conflict rule and compile-check the Darkroom bridge against the narrower façade.

- [ ] **Remove public items that are implementation details or have no production consumer.** `OutputAddress` is defined and re-exported but unused, `EventTrigger` and `RunEvent` are worker-internal transport, and execution-only graph validation helpers are public despite being called only by the compiler (`src/lib.rs:20-27`, `src/execution/identity.rs:22-26`, `src/execution/event.rs:19-25`, `src/execution/report.rs:31-35`, `src/graph/validate.rs:274-295`). Delete `OutputAddress` unless Batch 1 adopts it as the canonical public address, stop re-exporting internal transport, and narrow `validate_for_execution`/its debug wrapper to `pub(crate)`. Audit the rest of `lib.rs` with workspace-wide usage plus an API diff, then compile every workspace consumer.

- [ ] **Remove the direct `strum` and `uuid` dependencies from Scenarium.** `strum` exists only for the blanket `EnumVariants` bridge and `TypeEntry::enum_of`; the sole non-Scenarium caller needing that bridge can pass `VariantNames::VARIANTS` explicitly from Lens, which already depends on `strum` (`src/data/type_system.rs:5-21`, `src/library.rs:95-102`, `../lens/src/image/nodes/mod.rs:24-32`). `uuid` is used only by an otherwise unused `TypeId::from_name` helper and hyphen-free cache filenames; delete the unused helper and format `NodeId::as_u128()` directly for filenames (`src/data/type_system.rs:24-30`, `src/execution/disk_store/mod.rs:85-100`). Confirm both crates disappear from `cargo tree -p scenarium --depth 1` and that cache path uniqueness/format remains exact.

- [ ] **Standardize on standard-library hash maps unless the faster hasher is measured as necessary.** The crate mixes `std::collections` and `hashbrown` even within `Library`, `graph::wiring`, and graph cloning, creating aliases solely to reconcile map types; no production code uses a hashbrown-only API (`src/library.rs:1-11`, `src/graph/wiring.rs:1-4`, `src/graph/clone.rs:1-4`, `Cargo.toml:20`). Replace it with `std::collections::{HashMap, HashSet}` and remove the direct dependency, then benchmark compile/plan/resolve on a large graph because those node-keyed maps are hot. If the hasher difference is material, keep `hashbrown` consistently and document the measured reason rather than retaining the current mixture.

## Open questions

- [ ] **What filesystem identity guarantee do file-reading functions require?** The implementation fingerprints files by `(length, mtime)` and directories by only their immediate entries' metadata, while the docs mention an opt-in content hash that does not exist in the code (`src/execution/resource/mod.rs:24-43`, `src/execution/resource/mod.rs:64-128`, `src/execution/digest/mod.rs:22-29`, `src/execution/README.md:195-204`). Same-size edits within timestamp granularity—and some nested-directory content edits—can therefore reuse stale results. If that tradeoff is not acceptable for astronomical inputs, add an explicit resource fingerprint policy at the function-input contract and test metadata, recursive, and content-hash modes.

- [ ] **Should function/editor metadata remain inseparable from executable registration?** `Func`, `FuncInput`, and `FuncOutput` mix execution contracts with category, descriptions, display labels, defaults, and picker variants (`src/node/definition.rs:20-69`, `src/node/definition.rs:151-212`). Keeping one structure is ergonomic and ensures the editor sees exactly what executes; splitting declaration metadata from executable attachments would narrow compiler/cache dependencies but risks parallel schemas. Decide this only after the registry and context simplifications show whether a natural `FuncDecl`/`FuncImpl` boundary remains.

Recommended execution order is the batch order above. Within Batch 2, implement transparent retry and off-thread availability collection together; they want the same pre-execution disk snapshot. Batch 3's type-entry redesign should precede codec-registry extraction, and Batch 4's context split should precede the lambda ABI rewrite so `Invocation` is designed around the final lifecycle boundaries.
