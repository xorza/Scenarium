# Darkroom architecture and simplification review

## Executive summary

Darkroom has a strong broad shape: `Document` is the persisted source of truth, `Workspace` coordinates it with runtime services, edits flow through explicit intents and undo steps, and `Scene` is a read-only render projection. The main weaknesses are at the seams where delayed work is later interpreted against mutable “current” state, and where ostensibly frontend-neutral code depends on GUI navigation or preference types.

The highest-risk issues are:

- A bare `GraphRef::Local(GraphId)` cannot identify nested local graphs in Scenarium's recursive graph model.
- Worker reports have no request/document identity, and repeated graph instances lose their runtime occurrence identity.
- Undoing graph creation/detachment can remove a local graph while leaving its persisted view behind, producing a document that validation and save reject.
- Connection rules are duplicated in the canvas while the authoritative intent path accepts dangling or otherwise invalid bindings.
- Delayed tab and node-menu actions resolve positional or mutable selection state, so intervening edits can retarget them.
- New, Load, and script shutdown bypass the only unsaved-document transition policy.
- Script resource caps do not bound host-side output/effect buffers or outbound replies.

The largest simplifications are replacing the flat indexed dock tree with a recursive tree, making graph/runtime outcomes named rather than boolean, moving graph construction and interface semantics to the core crates that own them, and reorganizing the mutually dependent `gui/canvas` and `gui/node` trees as one graph-editor subsystem. No direct dependency is entirely unused, but the single-entry ZIP document format, mandatory script compression, unconditional GUI/scripting compilation, and duplicate Tokio runtimes all present concrete dependency or resource reduction opportunities.

## Current flow

`App` and `TerminalSession` both own a `Workspace`; `Workspace` pairs an `OpenDocument` with `RuntimeHost`. GUI edits are collected by `Editor`, converted from `Intent` to reversible `UndoStep`, applied to `Document`, and packed into `ActionStack`. Before rendering, `Scene` projects the active graph and its `GraphView` into flat retained pools. `RuntimeHost` compiles a graph, sends it to `WorkerBridge`, and frontends later drain untagged `WorkerEvent`s into GUI `RunState` or terminal status output.

## Batch 1 — Critical: preserve graph identity and enforce graph invariants

- [ ] **Make local-graph identity valid for the recursive graph model.** `GraphRef::Local(GraphId)` is resolved only through the root graph's `graphs` map, but adding a shared graph while editing a local graph inserts the copied definition into that active graph's own nested map (`src/core/document/mod.rs:307-353`, `src/core/document/mod.rs:394-410`, `src/core/edit/intent/apply.rs:138-165`). The badge still emits only the bare ID and opening silently stops when root lookup fails (`src/gui/node/prepass.rs:27-41`, `src/gui/app/editor/mod.rs:494-506`). Because `fresh_copy` preserves nested graph IDs, two instances can also contain identical IDs (`../scenarium/src/graph/clone.rs:54-70`). Prefer making every `GraphId` globally unique within a document: recursively remap copied IDs and `GraphLink::Local` references, validate uniqueness, and resolve graphs/scopes recursively. If parent-scoped identity is intentional, replace `GraphRef` with a structural graph path instead. Validate create/open/edit/undo at two nesting levels and two copies containing the same source nested graph.

- [ ] **Make execution requests, reports, and runtime occurrences carry complete identity.** `RuntimeHost::compile` overwrites one shared `flatten_map` on every compile—including cache-only compilation—while `WorkerEvent` carries neither a run ID nor the map, and `App` projects every later report through whichever map is current (`src/core/runtime_host.rs:36-39`, `src/core/runtime_host.rs:131-145`, `src/core/runtime_host.rs:190-197`, `src/core/worker.rs:24-35`, `src/gui/app/mod.rs:120-159`). A second request or document replacement can therefore attribute an older report to the wrong editor. Repeated instances add another collision: run-to-node accepts a bare `NodeId` and selects the first `NodeAddress` encountered in unspecified graph iteration order, while Darkroom collapses address-bearing pinned outputs back to `OutputPort(NodeId, index)`, making the displayed value completion-order dependent (`src/core/workspace/mod.rs:75-77`, `src/core/runtime_host.rs:175-185`, `src/gui/pinned_output.rs:57-68`, `../scenarium/src/execution/compile.rs:42-50`, `../scenarium/src/execution/identity.rs:69-90`). Introduce an atomic `ExecutionRequest { run_id, document_generation, compiled, flatten_map, seeds }`, echo it on every event, preserve `NodeAddress`/`OutputAddress` through presentation, and define whether a seed targets one occurrence, all occurrences, or rejects ambiguity. Keep cache-save compilation out of visible run state and replace optimistic booleans with request-aware worker states. Validate overlapping requests, replacement during execution, and two instances with different inputs.

- [ ] **Centralize local-graph insertion/removal so the graph definition, view, and tabs change atomically.** `Document` stores `graph.graphs`, `local_views`, and layout tabs independently (`src/core/document/mod.rs:258-274`), but reverting `AddNode` and `DetachGraph` removes only the nested graph from an `EditScope` (`src/core/edit/intent/apply.rs:343-350`, `src/core/edit/intent/apply.rs:415-428`). `ensure_valid_layout` prunes dead tabs but never removes orphaned views (`src/core/document/mod.rs:413-452`), while validation rejects an orphaned `local_views` entry (`src/core/document/validate.rs:75-82`). Add document-level `insert_local_graph`/`remove_local_graph` operations used by forward and reverse edits; removal must close tabs and drop the matching view in the same operation. Validate add/open/undo and detach/open/undo, then require `Document::validate` and save to succeed after both sequences.

- [ ] **Put all connection validity in one authoritative graph API.** Canvas code independently checks const-only inputs, boundary passthrough, data types, and cycles, while the intent boundary checks only cycles and can accept a missing producer or invalid port after a mid-drag edit or from scripting (`src/gui/canvas/connection_ui.rs:403-487`, `src/core/edit/intent/apply.rs:28-47`, `src/core/edit/intent/build.rs:139-145`). Add a Scenarium/core `validate_binding(library, input, output) -> Result<(), BindingError>` that covers endpoint and port existence, input policy, type compatibility, boundary rules, and cycles. Use it both for snap feedback and authoritatively in `commit_intent_cascading`; the GUI should render the returned reason rather than reimplement policy. Validate every pairing class, source deletion between press/release, and non-GUI intents.

- [ ] **Capture semantic targets when delayed GUI actions are created.** Navigation applies undo/redo before scanning last-frame responses, but tab IDs and `DockOp::{ActivateTab, CloseTab}` use `(TabGroupId, index)`, so a stale click can target a new occupant (`src/gui/app/editor/mod.rs:323-349`, `src/gui/dock/mod.rs:82-128`, `src/core/document/dock.rs:151-168`). Node-menu picks similarly carry only an action and later apply it to whatever selection is then live (`src/gui/canvas/node_menu.rs:21-39`, `src/gui/app/editor/mod.rs:291-320`). Key tab widgets and operations by `TabRef`, keep semantic tab identity in drags, and capture `NodeMenuRequest { action, targets }` when the menu opens. Resolve liveness only at apply time. Validate reorder/close/undo before tab scan and selection changes while a node menu remains open.

## Batch 2 — Critical: bound external control and destructive transitions

- [ ] **Replace quit-only confirmation with one pending destructive-document transition.** New and Load immediately replace the editor/document, while only Quit consults dirty state (`src/gui/app/commands/file.rs:25-71`, `src/gui/app/mod.rs:64-67`, `src/gui/app/mod.rs:238-295`); script `Shutdown` calls `quit` directly (`src/gui/app/mod.rs:183-187`). Model `Option<PendingDocumentAction>` for Quit, New, Load(path), and the chosen remote-shutdown policy, and route all through one Save/Discard/Cancel state machine. Defer “Don't ask again” persistence until Save actually succeeds: today it is disabled before a cancelled/failed Save As, leaving a dirty document that the next close discards without prompting (`src/gui/app/mod.rs:264-273`, `src/gui/app/commands/file.rs:77-110`). Validate Cancel, failed Save, and cancelled Save As preserve the old document and confirmation preference.

- [ ] **Use one size-capped request collector and enforce the cap through the outbound frame.** Rhai caps its own strings/collections/operations, but every `print` grows an uncapped host `String`, every `apply` sends through an unbounded channel, and a panic can leak shared print state into the next request (`src/core/script/engine.rs:18-96`, `src/core/script/mod.rs:153-172`, `src/core/script/mod.rs:251-279`, `src/core/script/mod.rs:331-339`). The claimed bidirectional 1 MiB frame limit is only checked inbound; replies serialize an arbitrarily large nested result before `write_frame`, which checks only `u32` length (`src/core/script/tcp/mod.rs:52-55`, `src/core/script/tcp/mod.rs:388-395`, `src/core/script/tcp/mod.rs:437-443`). Collect print text, edits, run, shutdown, and result data per request under explicit byte/action budgets, publish one batch only after successful evaluation, and use a capped serializer plus compressed and uncompressed outbound limits. Validate print/apply floods, panic isolation, and replies immediately below/above the limit.

- [ ] **Make the edit target explicit and stop exposing the full internal `Intent` enum as the automation API.** Rhai's generic `apply`/`apply_all` deserialize every current and future `Intent`, including selection, viewport, and dock operations (`src/core/script/engine.rs:98-158`, `src/core/script/prelude.rhai:1-11`); terminal/headless then chooses the target from the persisted GUI dock's active tab (`src/core/terminal_session/mod.rs:103-114`, `src/core/document/mod.rs:360-365`). Define a narrow script edit DTO or service with an explicit `GraphRef`/graph ID, convert it to internal intents at one boundary, and keep GUI-only navigation variants private to the editor. This removes a hidden dependency on whichever tab a previous GUI session left active and makes headless behavior deterministic. Validate root and local-graph edits without any dock manipulation, plus rejection of missing targets.

- [ ] **Make script CLI configuration typed and fail fast.** Script flags are silently ignored without `--script-tcp`, and an invalid bind specification logs a warning then changes behavior to the default address (`src/main.rs:54-90`, `src/main.rs:110-121`). Parse a `BindSpec` through Clap and either require `--script-tcp` for dependent options or let any script option enable it; invalid network input should be a CLI error, not a fallback. Validate invalid bind, conflicting auth flags, dependent flags without enablement, and port/IP/full-address forms.

## Batch 3 — High: simplify state representations and make outcomes truthful

- [ ] **Replace the flat indexed dock tree with a recursive tree.** A depth-four cap bounds the layout to at most 31 nodes, yet the current representation serializes `NodeIdx` edges, maintains canonical preorder storage, repacks after structural changes, and validates reachability/index topology (`src/core/document/dock.rs:9-26`, `src/core/document/dock.rs:41-105`, `src/core/document/dock.rs:198-223`, `src/core/document/dock.rs:486-609`). Store `DockTree::Group(TabGroup) | Split { dir, ratio, first: Box<DockTree>, second: Box<DockTree> }`; structural equality then works directly and empty-child collapse becomes ordinary recursive replacement. Retain `DockPath` only as a transient divider address. Round-trip persisted layouts and sweep split/move/close/depth-limit behavior before deleting `NodeIdx`, canonicalization, and dead-slot validation.

- [ ] **Key sticky errors by operation family rather than exposing one public `Option<String>`.** `StatusLog` documents same-family clearing but any run, file, library, or compile success can assign `None` to the single slot and erase an unrelated failure (`src/core/status.rs:14-23`, `src/core/runtime_host.rs:107-118`, `src/core/runtime_host.rs:137-149`, `src/gui/app/mod.rs:126-149`). Successful load/save is worse: it calls `save_preferences`, which can set a new error, and then immediately clears it (`src/gui/app/commands/file.rs:55-72`, `src/gui/app/commands/file.rs:91-110`, `src/gui/app/commands/prefs.rs:52-58`). Store errors by a small `StatusDomain`, expose `clear(domain)`, and derive the displayed latest error without direct field writes. Validate that run success cannot clear file/preferences failure and successful project I/O cannot erase a failed preferences write.

- [ ] **Return a transactional named graph-publication outcome instead of `bool`.** The current boolean conflates an invalid target, library change, document-lineage change, and persistence failure, so the GUI knowingly marks update-in-place publication dirty (`src/core/runtime_library/mod.rs:43-47`, `src/core/runtime_library/mod.rs:78-99`, `src/core/runtime_host.rs:97-118`, `src/gui/app/commands/graph.rs:88-105`). Publication also mutates the document origin before durable graph-library persistence completes (`src/core/edit/publish.rs:10-39`). Stage the library/document changes, persist, then commit both (or explicitly retain a recoverable dirty library state), and return `PublishOutcome { library_changed, document_changed, persistence }`. Validate new publication, update-in-place, invalid target, and save failure; only the new-link case should dirty the document.

- [ ] **Replace coarse preference and terminal booleans with named effects/outcomes.** `PrefsCommand::Changed` makes a viewer filter/background edit rebuild the theme, clone the Aperture theme, check ML configuration, and save (`src/gui/main_window.rs:123-135`, `src/gui/app/commands/prefs.rs:16-49`); use `PersistOnly`, `ThemeChanged`, and `MlModelsChanged` or an effect set. `TerminalSession::save` returns `false` for both untitled and I/O failure, so TUI prints “nothing to save” after a real failure, and `run` prints “run queued” while ignoring compile failure (`src/core/terminal_session/mod.rs:61-74`, `src/tui/mod.rs:61-70`). Return `Result<SaveOutcome, _>` and a named run submission outcome. Validate exact side effects/messages for each variant.

- [ ] **Derive edit side effects from applied undo steps, not from whichever input path invoked them.** `commit_batch` centrally observes relayout, reconciliation, and dirty effects, but cache flushing is inferred earlier by scanning raw GUI intents (`src/gui/app/editor/mod.rs:123-176`, `src/gui/app/editor/mod.rs:403-425`). Stale/no-op intents can therefore flush, while script/external edits and undo/redo of cache changes do not (`src/gui/app/editor/shortcuts.rs:48-72`, `src/gui/app/mod.rs:339-341`). Add `UndoStep::effects() -> EditEffects` and aggregate the named effects in commit, undo, and redo. Apply each effect once after the transaction. Validate cache toggle/apply/undo/redo from GUI and script, plus stale intents that produce no step.

## Batch 4 — High: make persisted and retained state explicit

- [ ] **Separate configuration read outcomes from recovery mutations.** `Preferences::load` maps missing, unreadable, and malformed TOML to the same defaults, so a later normal save can overwrite a recoverable file; preferred-document load failure immediately clears and saves the path even for a transient missing drive or permission (`src/core/io/preferences.rs:148-155`, `src/core/workspace/mod.rs:21-42`). Graph-library quarantine also reuses one `.broken` target and can overwrite or fail on a second recovery (`src/core/io/graph_library/mod.rs:77-84`, `src/core/io/graph_library/mod.rs:124-134`). Consolidate reads around `LoadOutcome<T> { Missing, Loaded(T), Invalid/Unavailable(error) }`, never mutate preference intent during a read, and quarantine to unique non-clobbering names. Validate malformed preferences survive, a temporarily unavailable document path remains configured, and two quarantines retain both originals.

- [ ] **Scope persistent inspector state by graph rather than pruning it against the active scene.** `GraphUI` promises pinned inspectors survive navigation, but `Inspectors::apply` retains only IDs in the active `Scene`, so the first frame after A→B deletes every pinned inspector from A (`src/gui/canvas/inspector.rs:16-21`, `src/gui/canvas/inspector.rs:98-120`, `src/gui/canvas/mod.rs:70-83`, `src/gui/canvas/mod.rs:259-262`). Store `HashMap<GraphRef, PerGraphUiState>` and reconcile each graph against its own contents or the whole document; use deterministic insertion order if overlapping-panel paint/hit order matters instead of `HashMap` iteration (`src/gui/canvas/inspector.rs:141-162`). Validate A→B→A persistence, deletion while A is inactive, replacement of the document, and overlapping panels.

- [ ] **Give retained geometry an owner, liveness policy, and layout revision.** `CanvasGeometry` clears live maps but never invalidates or prunes persistent node-size and port-offset caches (`src/gui/canvas/geometry.rs:45-70`, `src/gui/canvas/geometry.rs:176-215`). Culling uses the cached size before a node can record again, so an off-screen node that grows into the viewport can remain permanently culled (`src/gui/node/mod.rs:144-153`, `src/gui/canvas/cull.rs:46-49`). Key caches by graph, attach a layout-affecting node revision/fingerprint, treat mismatches as unknown so the node records once, and retain only live node/port IDs. Validate bounded cache cardinality through create/delete cycles and off-screen growth across a viewport boundary.

- [ ] **Make image-viewer tabs output-only by construction.** `TabRef::ImageViewer` stores a general `PortRef`, while validation checks only node existence and allows input-kind or out-of-range viewer tabs (`src/core/document/mod.rs:52-83`, `src/core/document/mod.rs:282-287`, `src/core/document/validate.rs:84-90`). Resource retention ignores input tabs, but rendering converts the same index to an `OutputPort` anyway (`src/core/document/mod.rs:379-388`, `src/gui/main_window.rs:122-131`). Store `TabRef::ImageViewer(OutputPort)` and perform library-aware output-index liveness during normalization. Validate input-kind, out-of-range, removed-node, and valid-output round trips.

- [ ] **Stop using mutable boundary-port names as fallback identity.** After interface compaction, boundary-port lookup falls back from an index hint to the first matching name, while duplicate names are permitted (`src/core/document/mod.rs:290-305`, `src/core/document/mod.rs:497-528`, `src/core/edit/intent/types.rs:354-370`). Undo/redo can therefore rename the wrong same-named port. The smaller model is to enforce unique input names and unique output names in Scenarium validation; if duplicates are a real requirement, give interface ports stable IDs and use them in edit steps. Validate renaming the second duplicate, compacting an earlier slot, then undo/redo.

## Batch 5 — Medium: reduce frontend, runtime, and storage dependencies

- [ ] **Make core startup/configuration independent of GUI preferences.** The core module promises no Aperture dependency, but `core::io::preferences` stores `aperture::ImageFilter`, and `Workspace`/`RuntimeHost` accept the entire `Preferences` object merely to load a path and derive ML model paths (`src/core/mod.rs:1-5`, `src/core/io/preferences.rs:1-5`, `src/core/io/preferences.rs:67-83`, `src/core/workspace/mod.rs:47-62`, `src/core/runtime_host.rs:52-86`). Move viewer/window/theme preferences to GUI ownership, introduce a small serialized magnification enum mapped by `image_viewer`, and pass a `RuntimeConfig`/`MlModelPaths` plus an already selected `OpenDocument` into the core. Validate preference serialization and both filter mappings, then require `rg 'aperture' src/core` to find only prose links.

- [ ] **Feature-gate the GUI and scripting stacks after restoring those boundaries.** The single binary declares GUI, TUI, headless, and script modules together and all direct dependencies are unconditional (`src/main.rs:1-16`, `Cargo.toml:7-39`). Split a library plus small bin entry points or add explicit `gui` and `script-tcp` features; remove the direct `rhai`, `lz4_flex`, and `tokio-util` edges when scripting is disabled and desktop-only edges when GUI is disabled. Describe wins per edge rather than promising that each package disappears: `common` still depends on LZ4 and `imaginarium` still depends on `image`, while headless installs Lens image/astro functions (`../common/Cargo.toml:16-24`, `src/core/runtime_library/mod.rs:117-128`). Compare feature-specific `cargo tree`, clean build time, and release size.

- [ ] **Centralize background-runtime ownership without letting synchronous scripts starve execution.** `WorkerBridge` and optional `ScriptHost` each create a default multi-thread Tokio runtime and retain dead-read lifetime-holder fields even though `RuntimeHost` owns both (`src/core/background_runtime.rs:1-35`, `src/core/worker.rs:37-67`, `src/core/script/mod.rs:343-396`, `src/core/runtime_host.rs:24-46`). One `RuntimeHost`-owned runtime simplifies lifetime/drop order, but Rhai evaluation runs synchronously inside an async task and may intentionally take seconds (`src/core/script/mod.rs:285-329`, `src/core/script/engine.rs:18-23`). Consolidate only with a dedicated blocking pool/thread or equivalent scheduler isolation; otherwise retain a separately configured script executor. Validate small-core responsiveness, simultaneous scripting/evaluation, cancellation, and clean GUI/TUI/headless shutdown.

- [ ] **Store the current one-entry document as bounded JSON and remove `zip`.** `.darkroom` contains exactly one `document.json`, so loading spends substantial code validating archive topology, duplicate entries, overlap, entry kinds, and two sizes before parsing the only payload (`src/core/io/document.rs:1-16`, `src/core/io/document.rs:18-71`, `src/core/io/document.rs:74-147`, `src/core/io/document.rs:210-224`). With no compatibility requirement, serialize the validated `Document` directly through `common`'s JSON path and retain the existing byte limit and durable publication; this removes the direct `zip` dependency and most of `io/document.rs`. Validate extension checks, size boundary, malformed JSON, structural validation, durable overwrite, and round-trip.

- [ ] **Use Aperture's existing Winit ownership for system theme detection and remove `dark-light`.** Darkroom imports a portal/platform discovery stack for one `dark_light::detect()` call (`Cargo.toml:26`, `src/gui/theme.rs:307-315`), while Aperture already owns the Winit window/event boundary (`../aperture/src/host/winit/mod.rs:603-690`). Expose initial system theme and `ThemeChanged` through Aperture/WinitHost, map it to `ColorScheme::System`, and remove the direct dependency. Validate initial Light/Dark selection, a live OS theme change, and the unknown-platform fallback; compare the direct dependency tree before and after.

## Batch 6 — Medium: restore single-responsibility module boundaries

- [ ] **Move node creation semantics out of `NewNodeUi`.** The palette widget currently owns graph-copy lineage, fresh IDs, graph-instance construction, and default binding seeding, then repackages the result into `Intent::AddNode` (`src/gui/canvas/new_node_ui.rs:19-26`, `src/gui/canvas/new_node_ui.rs:120-133`, `src/gui/canvas/new_node_ui.rs:306-394`). Put pure `build_func_add`, `build_special_add`, and `build_graph_instance_add` constructors under `core/edit/intent` and return `Intent` directly; leave grouping, filtering, and click rendering in the GUI. Validate exact defaults, fresh IDs, origin/local links, and output intent payloads without Aperture.

- [ ] **Make Scenarium the canonical node-interface resolver and leave `Scene` as a projection.** `Scene::rebuild` re-matches every `NodeKind`, resolves functions/graphs/special nodes, synthesizes boundary specs, and derives semantic flags even though Scenarium already owns partial interface and port-type queries (`src/gui/scene.rs:258-478`, `../scenarium/src/graph/query.rs:13-73`). Add a public borrowed node-interface view in Scenarium, with Darkroom adding only its editor placeholder ports, and write events directly into Scene's retained pool instead of allocating a temporary `Vec` per node (`src/gui/scene.rs:273-316`, `src/gui/scene.rs:549-569`). Cross-check every node kind, missing definition, boundary placeholder, wildcard output, and event arity.

- [ ] **Turn `GraphUI` into the graph-editor façade and place the mutually dependent node cluster beneath it.** Top-level `gui/node` imports canvas types while canvas imports node scanners/renderers, and callers bypass both through public `MainWindow` fields and submodules (`src/gui/canvas/mod.rs:46-98`, `src/gui/node/mod.rs:12-19`, `src/gui/main_window.rs:31-58`). At the same time, canvas leaves accept an application-wide context although most need only `Theme`, and `Option<AppCommand>` is repeatedly overwritten across visible surfaces or implicitly wins over a simultaneous shortcut (`src/gui/app/mod.rs:24-43`, `src/gui/main_window.rs:75-136`, `src/gui/app/editor/mod.rs:236-264`). Nest `gui/node` under one `gui/graph` domain, split a narrow graph context from app lifecycle state, expose only façade methods, and return `FrameEffects { commands, intents, ... }` with explicit ordering rather than one lossy command slot. Remove duplicated key/value identity from `image_viewers`/`ImageViewer` and inject a root-owned `TypeAppearanceRegistry` instead of hard-coding Lens's image type in generic node rendering (`src/gui/main_window.rs:123-135`, `src/gui/image_viewer.rs:44-88`, `src/gui/node/port_color.rs:57-69`).

- [ ] **Relocate reusable primitives out of feature-owned files.** The image viewer imports generic pan/zoom state from `canvas::pan_zoom`, pin preview imports `Badge` from `node::header`, inspector imports elapsed formatting from the same header, and port coloring imports interpolation from wire rendering (`src/gui/image_viewer.rs:26`, `src/gui/canvas/pin_preview.rs:27`, `src/gui/canvas/inspector.rs:37`, `src/gui/node/port_color.rs:20-22`). Move them to neutral `gui/viewport`, `gui/widgets/badge`, `gui/format`, and color helpers before nesting the graph modules. Validate with dependency searches and the existing zoom/format behavior.

- [ ] **Split the remaining responsibility hubs after ownership moves.** `document/mod.rs` mixes reference types, views, edit scope, output-resource queries, local-graph lifecycle, and the root model (`src/core/document/mod.rs:28-274`, `src/core/document/mod.rs:307-530`); use `document/refs.rs` and `document/view/{mod.rs,auto_layout.rs,serde.rs}` with canonical imports. `script/mod.rs` mixes wire DTOs, effect transport, cancellable tasks, executor loop, result capture, and host lifecycle (`src/core/script/mod.rs:91-181`, `src/core/script/mod.rs:183-341`, `src/core/script/mod.rs:343-408`); split `protocol.rs`, `effects.rs`, `executor.rs`, `task.rs`, and `host.rs`. After deleting inactive theme surface, split `gui/theme.rs` into `theme/{mod.rs,palette.rs,aperture.rs}` and move invariant geometry scalars to constants or a static `ThemeMetrics` rather than copying them into every preset (`src/gui/theme.rs:7-35`, `src/gui/theme.rs:350-400`, `src/gui/theme.rs:840-870`). These should be move-only refactors after the preceding ownership changes, verified by the normal full suite.

## Batch 7 — Medium: remove duplicate hot-path work and transient allocations

- [ ] **Make breaker targeting persistent, deduplicated, and incremental for the gesture.** Four `Vec` target lists are cleared and every visible target is retested against every accumulated scribble segment each frame; cubic wires multiply that by their sample count (`src/gui/canvas/breaker.rs:109-163`, `src/gui/canvas/breaker.rs:219-238`). Pin rendering also probes the same output twice, and targets disappear if pan/zoom culls them before release (`src/gui/canvas/pin_ui.rs:248-251`, `src/gui/canvas/pin_ui.rs:307-309`, `src/gui/canvas/mod.rs:60-65`). Persist an `IndexSet<BreakTarget>` for the gesture, probe only newly appended segments, split pure geometry from probing, and resolve remove-node supersession when draining. Validate long-scribble scaling, duplicate probes, cross→pan away→release, and node-versus-edge supersession.

- [ ] **Remove easy steady-record allocations and their parallel-order invariants.** Visible nodes allocate `Vec<Track>` for repeated grid rows, enum/preset editors allocate `Vec<&str>` adapters, dock rendering allocates `Vec<TabLabel>` parallel to `group.tabs`, and drop classification repeatedly collects chip rectangles (`src/gui/node/port_row/mod.rs:57-88`, `src/gui/node/value_editor.rs:62-84`, `src/gui/node/value_editor.rs:122-145`, `src/gui/dock/mod.rs:232-295`, `src/gui/dock/strip.rs:157-183`). Add narrow iterator/repeated-track APIs to Aperture, resolve tab labels through a callback/iterator, and consume chip rect iterators or retained scratch. Validate exact layout/drop behavior and compare warmed-frame allocation counts.

- [ ] **Materialize full viewer images on visible demand, not for every open tab.** Before rendering, `Editor::frame` reconciles every persisted viewer tab and synchronously performs CPU conversion plus texture registration, including inactive/restored tabs (`src/gui/app/editor/mod.rs:209-215`, `src/core/document/mod.rs:379-387`, `src/gui/pinned_output.rs:72-102`, `src/gui/pinned_output.rs:125-132`). Replace the debug-only “Deferred is unreachable” temporal assumption with explicit `Pending | Resident | Failed` state, prepare pixels off the UI thread, register only the texture on it, and request full content from visible viewer panes. Retain resources directly with the document predicate rather than building a per-frame `HashSet`. Validate inactive preview-only tabs, responsive first activation, close-during-preparation, pinned-only output, and source removal.

## Open questions

- [ ] **Is the packed undo representation justified by measurements?** `ActionStack` spends substantial code serializing/deserializing every entry into a 1 MiB byte arena and maintaining byte ranges, a head offset, lazy compaction, and parallel metadata (`src/core/edit/action_stack/mod.rs:1-64`, `src/core/edit/action_stack/mod.rs:71-267`). If profiling does not show a material memory/allocation win on realistic large edits, replace it with an owned `VecDeque<HistoryEntry { target, steps }>` and a simpler approximate byte budget; otherwise document the benchmark that makes the complexity earn its place.

- [ ] **Does mandatory LZ4 compression help the local scripting protocol after both directions are bounded?** The protocol requires LZ4, but currently only requests enforce the advertised 1 MiB limit and replies allocate before framing (`src/core/script/tcp/mod.rs:17-29`, `src/core/script/tcp/mod.rs:388-443`, `Cargo.toml:31-35`). After Batch 2 adds a real outbound cap, measure representative loopback scripts and replies; if savings are negligible, use raw length-prefixed UTF-8/JSON and remove Darkroom's direct `lz4_flex` edge. The package remains until `common`'s independent codec use is also redesigned.

- [ ] **Is custom theme serialization an active product surface?** The UI exposes only System/Dark/Light, yet production theme types retain broad serde derives, field-order constraints, and a serialized Aperture theme (`src/gui/theme.rs:301-329`, `src/gui/theme.rs:346-428`, `src/gui/preferences_view.rs:57-79`). If serialization exists only to maintain the checked-in reference asset, replace it with an explicit asset-generation DTO or formatter and remove serde from the runtime theme model.

- [ ] **Do script clients need persistent Rhai scopes?** Each of up to 32 sessions retains an unrestricted `Scope` for ten idle minutes, so bounded request data does not bound retained state (`src/core/script/session/mod.rs:17-26`, `src/core/script/session/mod.rs:51-81`). If requests can be stateless, remove session IDs, the store, expiry policy, and persistent-scope memory entirely. If persistence is required, add an explicit retained-state budget/reset operation and verify a client cannot accumulate unbounded values across otherwise valid requests.

## Recommended execution order

Implement Batch 1 before structural refactors because graph identity and binding validity shape every later API. Batch 2 should precede scripting feature-gating. Batch 3's outcome types and Batch 4's ownership fixes should land before reorganizing graph GUI modules. Batch 5 creates the dependency boundaries needed for separate builds. Batches 6–8 can then proceed independently in small subsystem-focused changes.
