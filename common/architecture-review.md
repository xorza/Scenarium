# `common` architecture review

## Executive summary

`common` is structurally a leaf crate, but several of its APIs currently sit on data-integrity boundaries without enforcing the invariants their callers rely on. The highest-risk issues are unchecked introspection casts that turn editable negative integers into huge unsigned configuration values, `KeyIndexVec` combining ordered persistence, mutable authoring state, render projection, compile-time interning, and runtime dense indexing behind one unsafe key-in-value contract, and serializers/file scanners that can report success while producing unreadable or nondeterministic state.

The crate has also accumulated a second class of problem: many modules have exactly one production owner. Scenarium alone uses the Tokio synchronization wrappers, while Lumos alone uses the packed mask, coordinate/color types, FNV hasher, parallel helpers, and CPU dispatch. Moving those implementations to their owners would let `common` shed several dependencies and leave its published surface centered on genuinely cross-crate contracts such as serialization, typed IDs, graph containers, cancellation, spans, and shared debug/float utilities.

Current flow is: `common/src/lib.rs` publishes the common surface; Scenarium consumes graph containers and runtime primitives; Lens turns `Introspect` metadata into editable node inputs; Lumos consumes image/CPU/file helpers; and Darkroom persists documents, preferences, libraries, and undo steps through the serialization layer. The findings below follow those boundaries.

### `KeyIndexVec` replacement decision

| Candidate | Fit |
| --- | --- |
| Thin order-sensitive wrapper over `IndexMap<K, V, S>` | **Recommended for ordered keyed state.** It already uses a dense entry vector plus a hash table of entry indices, supports average-O(1) key/index lookup and index swaps, preserves capacity across `clear`, and stores the canonical key separately from the mutable payload. The wrapper is necessary because raw `IndexMap` equality is order-insensitive and its default Serde representation is a map; this project needs ordered equality and ordered-sequence serialization. |
| Plain `Vec<V>` plus a reusable key→index map | **Recommended only for runtime columns that become purely `NodeIdx`-addressed after reconciliation.** It keeps execution iteration/indexing minimal and lets the temporary reconciliation index stay out of the run path. |
| Sorted side `Vec<(K, usize)>` | Reject as the general replacement. It removes hashing but makes lookup O(log n), insertion/removal O(n), still needs a second allocation, and must update stored indices on every ordered edit. |
| One `Vec` with linear key search | Useful only for proven tiny collections. It has the lowest storage cost, but Scene and graph validation perform enough repeated lookup that it turns common passes quadratic. |
| `SlotMap` / slab / generational arena | Reject. Their generated transient handles do not replace persisted UUID identity or stable paint/serialization order; retaining UUID lookup requires a second map and gives more state, not less. |
| `BTreeMap` | Reject. It loses dense numerical indexing and performs per-entry allocation. |
| `ArrayVec` | Reject for authored graphs. It is literally heap-free but imposes a fixed document-size ceiling; it remains appropriate only for independently bounded scratch collections. |

A release-mode directional probe with UUID-shaped `u128` keys found successful lookup at roughly 7–8 ns for both the current structure and `IndexMap` with the standard hasher, versus 2.1–2.6 ns for `IndexMap` with hashbrown's fast builder over 8–4096 entries. At 1024 exact-capacity entries, the current duplicated-key layout requested 100,368 bytes while key-separated `IndexMap` requested 51,216; both performed zero allocations for `clear` plus refill within retained capacity. This is not a production benchmark, but it confirms that key separation and hasher choice are the worthwhile axes; a sorted or linear structure is not the default answer.

Implementation status updated 2026-07-18. Checked items are implemented; unchecked items remain open because they require broader ownership, persistence-policy, or schema decisions.

## Batch 1 — Critical: make introspection a checked value boundary

- [x] **Replace lossy numeric reflection with typed, checked conversions.** `FieldValue` has only `Int(i64)` and `Float(f64)` (`common/src/introspect/mod.rs:20-30`), while the derive advertises every integer width through `u128`/`i128` (`common/common-derive/src/lib.rs:129-166`), serializes defaults with `as i64` (`common/common-derive/src/lib.rs:210-235`), and rebuilds integers/floats with unchecked `as` casts (`common/common-derive/src/lib.rs:244-258`, `common/common-derive/src/lib.rs:283-305`). Lens feeds editable graph `i64` values into this path (`lens/src/config_node.rs:110-119`, `lens/src/config_node.rs:174-200`), and real reflected configs contain many `usize` fields (`lens/src/astro/configs.rs:49-56`, `lens/src/astro/configs.rs:98-105`, `lens/src/astro/configs.rs:146-153`), so `-1` currently becomes `usize::MAX`. Make `FieldKind` retain the concrete numeric domain, use `TryFrom`/range checks (including finite/range checks for `f64 -> f32`), and change `Introspect::from_fields` to return a typed `Result` so invalid authored input reaches the node error path. Either add lossless signed/unsigned `FieldValue` variants or explicitly stop advertising widths the `i64` graph value model cannot represent. Verify every exact min/max boundary, negative-to-unsigned input, and default round trip.

- [x] **Make `FieldValue::Null` actually clear optional fields.** `Null` is documented as an `Option` field's `None` (`common/src/introspect/mod.rs:27-30`), but generated option readers have no `Null` arm and fall back to `d.field` (`common/common-derive/src/lib.rs:283-315`); an `Option<T>` whose default is `Some(_)` therefore cannot be cleared. Generate an explicit `Some(FieldValue::Null) => None` path, preserve the default only for a missing field (and either error or deliberately default on a wrong kind), and verify `Default::Some` and `Default::None` across missing, null, valid, and mismatched inputs.

- [x] **Give introspected enums a collision-resistant explicit identity.** The derive records only the final syntactic type segment (`common/common-derive/src/lib.rs:136-166`) even though `FieldKind::Enum::type_name` is described as a stable identity (`common/src/introspect/mod.rs:40-43`); Lens hashes that short string into a `TypeId` and silently skips a second registration (`lens/src/config_node.rs:140-155`, `lens/src/config_node.rs:203-210`). Two modules that each define `Mode` would consequently share one wire type and whichever variant list registered first. Put an explicit stable key on `IntrospectEnum` (rather than deriving identity from field spelling), and make duplicate registration compare metadata and fail on disagreement. Verify same-named enums in different modules remain distinct and that renaming a display label does not change identity.

## Batch 2 — Critical: replace `KeyIndexVec` with key-separated storage

- [ ] **Retire `KeyIndexVec` and `KeyIndexKey` in favor of a key-separated, order-sensitive `IndexMap` wrapper.** The current type duplicates each key between `V` and `idx_by_key` and requires `items[idx_by_key[k]].key() == k` by convention (`common/src/key_index_vec.rs:9-20`), while `iter_mut`, `by_key_mut`, numeric `IndexMut`, mutable iteration, and `CompactInsert` expose unrestricted payload mutation (`common/src/key_index_vec.rs:147-176`, `common/src/key_index_vec.rs:244-253`, `common/src/key_index_vec.rs:284-315`, `common/src/key_index_vec.rs:345-355`). Replace it with a small `OrderedMap<K, V, S>` over `IndexMap`: the map key is canonical and immutable through value mutation; equality compares the ordered entry slice; Serde uses an ordered sequence and rejects duplicate keys; removal is explicitly stable; and `move_to_index` delegates to `move_index`. Store only payload in `V` where practical—`GraphView` can become `OrderedMap<ItemRef, Vec2>` and delete `ViewItem.key` (`darkroom/src/core/document/mod.rs:181-200`, `darkroom/src/core/document/view_item.rs:8-39`), while `SceneNode` can drop its mirrored `id` (`darkroom/src/gui/scene.rs:20-28`, `darkroom/src/gui/scene.rs:140-143`). For identity-bearing public values (`Node`, `Func`, `Graph`), either split the stored payload from its owned construction/snapshot form or narrow mutation so the duplicated identity is never exposed mutably; `Graph::iter_mut` and `find_node_mut` currently make `Node.id` writable after insertion (`scenarium/src/graph/mod.rs:180-183`, `scenarium/src/graph/mod.rs:292-341`). Verify order-sensitive equality, exact order across every Serde format, duplicate rejection, stable removal/reorder, and that no mutation path can change a stored identity.

- [ ] **Move compile interning and cache reconciliation out of the generic container.** `ExecutionProgram` is freshly default-constructed for every compile (`scenarium/src/execution/compile.rs:88-101`), so its `CompactInsert` is not preserving nodes across compiles; it only interns forward references encountered through `insert_with` (`scenarium/src/execution/flatten/mod.rs:72-108`, `scenarium/src/execution/flatten/mod.rs:305-310`, `scenarium/src/execution/flatten/mod.rs:497-510`). Use an `OrderedMap<NodeId, ExecutionNode, S>` entry operation for that first-encounter index and remove `ExecutionNode.id`. Runtime state has a different shape: `NodeIdx` is already the canonical address shared by the program, cache, and plan columns (`scenarium/src/execution/program.rs:21-53`), while `RuntimeCache` retains a full keyed container solely to realign slots on install (`scenarium/src/execution/cache/mod.rs:275-285`, `scenarium/src/execution/cache/mod.rs:342-353`). Make slots a plain `Vec<RuntimeSlot>` without a duplicated id, reconcile it before replacing the old program using the old/new ordered key slices plus reusable scratch, then run exclusively by `NodeIdx`. Verify producer-before-emitter forward references keep the correct index, cache state survives every add/remove/reorder permutation by id, removed slots are dropped, new slots default, and execution performs no UUID lookup once a seed/event has resolved to `NodeIdx`.

- [ ] **Define “allocation-free” as a capacity contract and enforce it at hot boundaries.** An unbounded persisted graph cannot be literally heap-free: both the current `with_capacity` and the proposed ordered hash map allocate storage and must grow after their capacity is exceeded (`common/src/key_index_vec.rs:57-61`). Promise instead that lookup, iteration, mutation within capacity, Scene rebuild, runtime reconciliation, planning, and execution allocate nothing in the container; reserve at cold boundaries from known sizes, retain capacity on clear, and add counting-allocator checks for the Scene rebuild and cache reconcile paths. Compilation is a separate ownership issue: it creates a new program that is moved to the worker on every call (`scenarium/src/execution/compile.rs:88-114`), so making compile itself allocation-free requires recycling returned program buffers or maintaining an explicit pool, not choosing another map. Benchmark representative real graph sizes with the standard and a named fast hasher; if untrusted documents/scripts are considered adversarial, keep the HashDoS-resistant builder at that boundary or cap/validate collection sizes rather than silently trading resilience for the synthetic lookup win.

- [x] **Make equality include the stable iteration order.** `PartialEq` performs key lookup and deliberately ignores item order (`common/src/key_index_vec.rs:22-35`), although the container offers stable reordering (`common/src/key_index_vec.rs:100-118`) and serializes `items` in order (`common/src/key_index_vec.rs:363-393`). That makes `Graph` equality ignore an order its own API calls load-bearing (`scenarium/src/graph/mod.rs:292-299`) and makes `GraphView` equality ignore its persisted, undoable paint stack (`darkroom/src/core/document/mod.rs:172-201`). Replace the lookup-based comparison with ordered `items` equality. Verify differently ordered `KeyIndexVec`, `Graph`, and `GraphView` values compare unequal while serialization round trips preserve equality.

## Batch 3 — High: make every serializer output readable by its paired reader

- [ ] **Apply the LZ4 decode ceiling during encode and make its length conversion checked.** Encoding writes `temp_buffer.len() as u32` without checking the reader's 1 GiB ceiling (`common/src/serde.rs:40-51`, `common/src/serde.rs:82-100`), so a large value can serialize successfully into a payload this crate refuses to read, and a payload above `u32::MAX` wraps its header. Move the maximum to a shared constant, reject oversized JSON bytes before allocating/compressing, and use `u32::try_from`. Exercise the length check at the ceiling, ceiling plus one, and `u32::MAX + 1` through a small helper rather than allocating multi-gigabyte fixtures.

## Batch 4 — High: make filesystem discovery and persistence deterministic and durable

- [x] **Return directory-scan errors and sort successful results.** `files_with_extensions` turns `read_dir` failure into an empty vector, drops individual entry errors, and treats metadata failures as non-files (`common/src/file_utils.rs:12-33`), so Lens reports an existing unreadable folder as merely empty (`lens/src/astro/library/mod.rs:266-284`). The result order is also filesystem-defined even though the reference input is documented as an index into directory-sorted lights (`lens/src/astro/library/mod.rs:241-243`). Return `io::Result<Vec<PathBuf>>`, propagate entry/metadata failures with path context, and sort by filename/path before returning. Verify missing, unreadable, readable-empty, and populated directories remain distinct and ordering is stable.

- [x] **Move extension policy beside the decoder that owns it.** `common::file_utils::astro_image_files` combines RAW and FITS extensions (`common/src/file_utils.rs:7-10`, `common/src/file_utils.rs:36-43`), but both directory consumers feed those paths into RAW-only CFA loading: `CfaImage::load` always calls `load_raw_cfa` (`lumos/src/io/astro_image/cfa.rs:62-83`) and `calibrate_align_stack` explicitly expects raw lights (`lumos/src/stacking/pipeline/streaming.rs:35-63`). Conversely, Lens's general loader accepts 13 RAW/FITS/standard formats (`lens/src/astro/library/mod.rs:38-45`) while its directory error claims all those categories even though the common helper excludes standard images (`lens/src/astro/library/mod.rs:275-284`). Keep only a generic fallible sorted scanner, place the authoritative RAW set beside Lumos's RAW decoder and the full loadable set beside `AstroImage::from_file`, and ensure each caller selects only formats its downstream decoder accepts.

- [ ] **Publish durable serialized state with atomic replacement.** Document, library, preference, and CFA-master saves overwrite their final paths directly (`darkroom/src/core/io/persistence.rs:61-66`, `darkroom/src/core/io/library.rs:103-116`, `darkroom/src/core/io/preferences.rs:178-185`, `lumos/src/io/astro_image/cfa.rs:112-116`), so interruption or disk-full can destroy the last good copy. The workspace already has two sibling-temp-plus-rename implementations, including a concurrency-safe cleanup-aware version (`scenarium/src/execution/disk_store/mod.rs:247-303`) and a simpler duplicate (`darkroom/src/core/script/tcp/mod.rs:202-220`). Consolidate a uniquely named, platform-correct byte publication helper in the shared file layer; offer explicit durable (`sync_all` as required) and cache/non-durable modes, then route authored state through the durable one. Verify replacement, concurrent writers, temp cleanup, and preservation of the prior file when write/publication fails.

- [ ] **Stop using `FnvHasher` as persistent frame-cache identity.** `FnvHasher` is a fast deterministic non-cryptographic hasher (`common/src/fnv.rs:1-34`), but Lumos uses only its 64-bit path hash as the cache filename (`lumos/src/stacking/frame_store/mod.rs:325-329`) and validates reuse with plane byte length plus a sidecar containing whole-second mtime (`lumos/src/stacking/frame_store/mod.rs:336-341`, `lumos/src/stacking/combine/cache/loader/mod.rs:324-371`, `lumos/src/stacking/combine/cache/loader/mod.rs:390-414`). A collision between same-sized/same-mtime inputs reuses the wrong pixels, and rewriting one source twice within a second can reuse stale data without any collision. Use a collision-resistant full digest for persistent filenames and store/compare source identity, size, and high-resolution modification data in the sidecar; keep FNV only for the transient spatial hash. Verify a forced cache-key collision and a same-path rewrite within one second.

## Batch 5 — Medium: put Scenarium runtime primitives in Scenarium

- [ ] **Move `Shared`, `Slot`, `PauseGate`, and `ReadyState` to Scenarium and remove Tokio/arc-swap from `common`.** `Shared` has one production consumer that immediately wraps it as `SharedAnyState` (`common/src/shared.rs:5-28`, `scenarium/src/runtime/shared_any_state.rs:8-36`); `Slot`, `PauseGate`, and `ReadyState` likewise serve only Scenarium's event worker (`scenarium/src/elements/worker_events_library.rs:48-104`, `scenarium/src/worker/mod.rs:94-105`, `scenarium/src/worker/event_loop.rs:23-58`). Keeping them in the leaf crate makes every common consumer compile Tokio and arc-swap (`common/Cargo.toml:21`, `common/Cargo.toml:29`) for a single runtime owner. Move the primitives and their callers into Scenarium, then remove the exports and dependencies. Verify Scenarium standalone behavior and confirm unrelated common consumers no longer receive those runtime dependencies.

- [ ] **Make `PauseGate` close guards composable while moving the primitive.** Every `close()` stores one boolean (`common/src/pause_gate.rs:49-57`), and dropping any guard clears it and wakes all waiters (`common/src/pause_gate.rs:61-70`); two overlapping guards therefore reopen the gate when the first drops even while the second remains alive. Replace the boolean with a close count and notify only on the `1 -> 0` transition, or enforce exclusive close ownership with a clear invariant. Verify nested/concurrent guards dropped in both orders. The current worker holds only one serialized guard (`scenarium/src/worker/mod.rs:179-195`), so this repairs the public primitive rather than an observed worker failure.

- [ ] **Reduce `Slot` to its production send/observe contract.** Production uses only latest-value publication and `peek`/`peek_async` (`scenarium/src/elements/worker_events_library.rs:48-55`, `scenarium/src/elements/worker_events_library.rs:82-104`), while `SlotError`, `take`, `take_async`, `try_take`, and `has_value` add ownership-transfer semantics where a live peek makes taking fail (`common/src/slot.rs:6-8`, `common/src/slot.rs:47-88`, `common/src/slot.rs:109-115`). Delete the unused half when relocating the type; verify frame-event publication still overwrites the old value and wakes every waiting observer.

## Batch 6 — Medium: return Lumos-only implementation to Lumos

- [ ] **Move the image-domain value types out of `common`.** `BitBuffer2`, `Vec2us`, and `Rgb` have no production consumer outside Lumos; representative owners are the star-detection buffer pool (`lumos/src/stacking/star_detection/buffer_pool.rs:6-23`), raw/image geometry (`lumos/src/io/raw/mod.rs:878-911`, `lumos/src/math/rect/mod.rs:49-67`), and image operations (`lumos/src/image_ops/mod.rs:32-59`). Relocate them into coherent Lumos modules, remove their root exports (`common/src/lib.rs:39`, `common/src/lib.rs:51`, `common/src/lib.rs:58`), and remove `aligned-vec` from `common/Cargo.toml:17`. This keeps image representation decisions near the SIMD/algorithm callers that constrain them.

- [ ] **Fix empty iteration and hot checks as part of the `BitBuffer2` move.** The type declares either zero dimension empty (`common/src/bit_buffer2.rs:126-136`), but `BitIter::next` checks only height before calling `get_xy(0, 0)` (`common/src/bit_buffer2.rs:324-346`), so `0 x N` iteration and both `Vec<bool>` conversions panic. Its per-bit `get`/`set` methods also use release assertions (`common/src/bit_buffer2.rs:138-178`) even though iteration pays them for every pixel, contrary to the project hot-path rule. Track a linear logical index (or terminate on either zero dimension), use `debug_assert!` in per-bit accessors, and keep release validation at cold operation boundaries. Verify `0 x N`, `N x 0`, and `0 x 0`, exact iterator lengths, conversions, scalar/SIMD agreement, and debug out-of-bounds behavior.

- [ ] **Move remaining Lumos-only utilities and delete the callback wrapper.** `FnvHasher`, CPU dispatch, and `try_par_map_limited` are production-used only by Lumos (`lumos/src/stacking/frame_store/mod.rs:14`, `lumos/src/stacking/star_detection/detector/stages/filter.rs:9`, `lumos/src/math/sum/mod.rs:24-68`, `lumos/src/stacking/combine/cache/loader/mod.rs:193-205`); the infallible `par_map_limited` has no non-benchmark production caller (`common/src/parallel.rs:22-46`). Move/prune those utilities, using Imaginarium's existing detector where its standalone boundary permits (`imaginarium/src/cpu_features.rs:1-55`). `SharedFn` merely recreates `Option<Arc<F>>` (`common/src/shared_fn.rs:3-39`) for the sole `ProgressCallback` alias (`lumos/src/stacking/progress.rs:25-35`), so replace it with a domain-local optional callback and delete the wrapper. This removes `rayon` from `common/Cargo.toml:20` and sharply reduces the public utility surface.

## Batch 7 — Low: remove avoidable copies and build dependencies

- [ ] **Give slice deserialization a true zero-copy dispatch path.** `deserialize(&[u8], ..)` wraps the slice in `Cursor` and then Bitcode copies the whole payload into scratch via `read_to_end` (`common/src/serde.rs:60-83`). Darkroom's hot undo path repeats that copy from an already contiguous action range (`darkroom/src/core/edit/action_stack/mod.rs:260-278`), contradicting its single-buffer/no-copy design (`darkroom/src/core/edit/action_stack/mod.rs:1-20`). Dispatch slices directly to each backend's slice decoder, reserve `deserialize_from` for actual readers, and update undo to call the slice path. Verify identical decoding/trailing-data behavior and that repeated undo/redo does not grow an O(payload) scratch buffer.

- [ ] **Replace the exported normalization trait with a private consuming helper.** `NormalizeString` is exported (`common/src/lib.rs:48`) but its only production use is internal TOML output (`common/src/serde.rs:7`, `common/src/serde.rs:38-40`); that call already owns a `String`, yet the no-CR path clones it (`common/src/normalize_string.rs:32-40`). Replace it with `fn normalize(mut value: String) -> String`, append the final newline in place on the fast path, allocate only when rewriting CR/CRLF, and remove the trait/re-export. Verify all existing line-ending semantics and allocation reuse for already-normalized TOML.

- [ ] **Compile the introspection derive only for the consumer that uses it.** `common-derive` is unconditional (`common/Cargo.toml:16`, `common/src/introspect/mod.rs:81`) even though Lens is the only production derive consumer (`lens/src/astro/configs.rs:49-466`), and the workspace enables Syn's broad `full` parser (`Cargo.toml:71`) while the derive only needs structs/enums, fields, types, and attributes (`common/common-derive/src/lib.rs:18-21`). Put the macro re-export behind an `introspect-derive` feature enabled by Lens and narrow Syn's features. Verify the traits/value types remain available without the feature, Lens derives still expand, and `cargo tree` for non-Lens consumers no longer includes `common-derive` solely through `common`.

## Open questions

- [ ] **Is `CancelToken::reset` guaranteed to run only after every clone from the previous operation has quiesced?** A live token's clones share one flag (`common/src/cancel_token.rs:23-32`), so `reset` makes any still-running old clone observe “not cancelled” again (`common/src/cancel_token.rs:46-64`). The current worker awaits a run before reset (`scenarium/src/worker/mod.rs:182-195`), which appears safe if all blocking work has joined. If quiescence is a hard contract, narrow reset ownership and document/assert it; if operations may overlap, issue a fresh per-operation flag so old tokens remain cancelled permanently.
